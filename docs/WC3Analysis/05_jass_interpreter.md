# 05 — JASS Interpreter

## What JASS Is

JASS (Just Another Scripting Syntax) is Blizzard's embedded scripting language for WC3 maps. Every map's game logic runs in JASS:
- Unit creation, orders, damage
- Trigger registration and callbacks
- Win/loss conditions
- Camera control
- Every single gameplay event

Without a working JASS interpreter you can load terrain and see the map, but nothing will happen — no units will move, no events will fire.

## Parser

```
jassparser/src/com/etheller/warsmash/parsers/jass/
```

The parser is generated by **ANTLR 4.7** from a grammar file. The generated output:
- `SmashJassLexer.java` — tokenizer
- `SmashJassParser.java` — LALR(1) recursive descent parser

**Entry point for parsing a map script:**
```java
// War3MapViewer.java (approximate)
InputStream jassStream = dataSource.getResourceAsStream("war3map.j");
JassProgram program = SmashJassParser.parse(jassStream);
```

The parser produces an AST made of classes in:
```
core/src/com/etheller/interpreter/ast/
  definition/   — function definitions, variable declarations
  statement/    — if/else, loop, exitwhen, return, set, call
  expression/   — arithmetic, comparison, function call, array index
  value/        — runtime value types
  scope/        — execution scopes
  function/     — function invocation
```

## Type System

JASS has a small, fixed type system:

| JASS type | Java class | Notes |
|-----------|-----------|-------|
| `integer` | `JassInteger` (boxed int) | 32-bit signed |
| `real` | `JassReal` (boxed float) | 32-bit float |
| `boolean` | `JassBoolean` | |
| `string` | `JassString` | Java String wrapper |
| `code` | `CodeJassValue` | Function pointer (integer index) |
| `handle` | `HandleJassValue` | Base for all game objects |
| `unit`, `player`, `trigger`, `timer`, … | subclasses of `HandleJassValue` | All are handles |
| `null` | `JassNull` | The null handle |

**Handle allocation:**  
Handles are reference-counted integers. `CSimulation` maintains an `int nextHandleId`. Every game object (`CUnit`, `CTrigger`, `CTimer`, etc.) gets a unique handle id. The JASS runtime resolves handles back to Java objects via `CSimulation.getObject(handleId)`.

## Execution Model

### GlobalScope

```
core/src/com/etheller/interpreter/ast/scope/GlobalScope.java
```

The global scope holds:
- All global variable values (`Map<String, JassValue> globals`)
- The function table (`Map<String, JassDefinitionBlock> userFunctions`)
- The native function table (`JassNativeManager nativeManager`)
- The thread queue for trigger execution

### Threads

JASS is **single-threaded from the game's point of view** but uses "virtual threads":
- Each trigger action runs in its own `Thread` object (not an OS thread)
- Threads are executed cooperatively inside `CSimulation.update()`
- `TriggerSleep` suspends the current thread until a timer fires
- `GlobalScope.runThreadUntilCompletion(thread)` runs the thread to its next yield/return

```java
// Simplified trigger execution flow
void fireTrigger(CTrigger trigger) {
    if (!trigger.evaluateConditions()) return;
    JassThread thread = globalScope.createThread(trigger.actionFunctionPtr);
    pendingThreads.add(thread);
}

// In simulation.update()
for (JassThread thread : pendingThreads) {
    globalScope.runThreadUntilCompletion(thread);
}
```

### Stack Frames

Each function call pushes a `DefaultScope` (local variable frame) onto a stack:
```java
class DefaultScope {
    JassDefinitionBlock function;
    Map<String, JassValue> locals;
    JassValue returnValue;
    int instructionPointer;
}
```

## Native Function Registry

```
core/src/com/etheller/warsmash/parsers/jass/JassNativeManager.java
```

Native functions are Java methods registered by name. The JASS runtime calls them when the interpreter encounters a `call NativeName(args...)` instruction.

**Registration pattern:**
```java
nativeManager.register("CreateUnit", (args, scope) -> {
    CPlayer player = (CPlayer) args[0].getHandle();
    String typeId   = args[1].getString();
    float x         = args[2].getReal();
    float y         = args[3].getReal();
    float facing    = args[4].getReal();
    CUnit unit = simulation.createUnit(player, typeId, x, y, facing);
    return new HandleJassValue(unit.getHandleId());
});
```

**The full native function list** is defined in:
```
core/src/com/etheller/warsmash/parsers/jass/scope/CommonTriggerExecutionScope.java
core/src/com/etheller/warsmash/parsers/jass/JassAIEnvironment.java
```

These map almost 1:1 to Blizzard's `common.j` declarations. For a reimplementation, `common.j` (inside `War3.mpq` at `Scripts/common.j`) is the authoritative list of all native function signatures.

## Initialization Sequence for a Map

```
1. Parse common.j          (native declarations — defines the API)
2. Parse blizzard.j        (BJ library — utility wrappers)
3. Parse war3map.j         (map-specific user code)
4. program.initialize()
     a. Topologically sort library blocks
     b. Define all functions in order
     c. Run auto-generated "InitGlobals" function (set global defaults)
5. Call "main()"           (map init code — registers triggers, creates units)
6. Call "config()"         (player/team setup for lobby)
```

**Execution entry point (approximately):**
```java
JassProgram program = ...;
program.initialize(globalScope);
globalScope.callFunction("main");
```

## Implementing a JASS Interpreter from Scratch

Minimum viable interpreter for running WC3 maps:

1. **Lexer:** tokenize keywords (`function`, `takes`, `returns`, `local`, `set`, `call`, `if`, `then`, `else`, `elseif`, `endif`, `loop`, `exitwhen`, `endloop`, `return`, `globals`, `endglobals`, `native`, `constant`), identifiers, integer/real/string literals, operators.

2. **Parser:** recursive descent is simplest. JASS grammar is LL(1) — no ambiguity.

3. **Runtime values:** use a tagged union or sealed class hierarchy for the 4 primitive types + handle.

4. **Native bridge:** a `HashMap<String, NativeFn>` where `NativeFn` is `(List<Value>, GameState) -> Value`. Register ~300 natives.

5. **Handle table:** `HashMap<Int, GameObj>` in the game state. Every handle is just an int that maps to a Java/Rust/Go object.

6. **Trigger execution:** a queue of pending function calls, drained each simulation tick.

7. **TriggerSleep / Wait:** implement as a "resume at tick N" record. Store in a priority queue sorted by resume tick.

**Shortcuts for reimplementation:**  
- Start with `common.j` + `blizzard.j` from the MPQ as your native function spec  
- Warsmash's `CommonTriggerExecutionScope.java` shows the Java signatures for every implemented native  
- Many "natives" in `blizzard.j` are pure JASS (no native body) — you get them for free if you run the file
